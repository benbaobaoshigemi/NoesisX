# 职责区分
## 总则
    整体架构采用 1+4+N 架构。一个代理（agent），四个固定子代理（sub-agent），任意创建用完即弃的若干临时子代理（temp-sub-agent）。agent 不得任意唤起 sub-agent，被 sub-agent 能力栈完整包含的周边工作也不宜由 agent 直接插手，必须按协议和场景使用。temp-sub-agent 如果有需求agent 和 sub-agent 都可以任意使用。

## 能力划分
- “1“指母代理（agent）。agent 直接在聊天窗口和用户对话。agent 承担主力任务完成和几乎全部的探索研究等工作。是团队的核心。需要借助子代理的力量独立完成研究并与用户良好交互。
- “4“涵盖四类固定子代理（sub-agent）。sub-agent 拥有独立的.toml配置文件,由agent按照协议在需要时实例化，可以用完即弃，也有长期保留的资格。
    - fetcher：fetcher 用于进行前端交互获取数据或信息，存储到 library 文件夹并按照规范命名。拥有相对独立的 skill 能力栈（前端交互相关 skill 对科研项目而言大多数时候是多余的，但是总有一些数据需要网页爬取而非直接下载）。
    - engineer：engineer 解决项目中纯粹的技术性工作，维护研究系统的顺畅运行。engineer 不涉及研究工作。环境混乱，依赖缺失，字体乱码等问题。它的意义是确保无价值的 debug 过程这样的垃圾工作不会干扰 agent 的研究进度。engineer 需要维护自己的一个文档。
    - writer：writer 负责撰写正式的报告文件并进行润色，确保内容准确，语言得当，格式美观。产物通常是 word 或者 pdf 格式。writer 也拥有相对独立的 skill 能力栈，用于撰写，润色，渲染与反思。
    - professor：professor 是最特殊的 sub-agent，尽管它是受到 agent 调用的子代理，但是 agent 调用它的用意通常是“请教“而非指派。professor 站在更高层级，判断当前整个工作区/当前研究的方向，甚至是更重要的————意义。尽管我强调各级代理都要善用联网检索，但是professor 必须更积极地实时多轮联网检索，避免闭门造车，给出关键指示。
- “N“指的是临时子代理（temp-sub-agent），应当被视作有智力的工具而非团队成员。agent 和 sub-agent 可以任意创建，指派明确的工作，完成任务之后立刻进行销毁。最常见的场景是临时对抗性审查，如思路审查，文字润色等工作。也可以进行多条路线的发散性探索（这里需要澄清：探索完成之后，研究状态仍有 agent 持有）。其他需要引入并发或者第三方独立者的场景也可以调用。


# 文件系统
## 总则
- 科研系统讲究可溯源，可复现。对话窗口具有天然的不可靠性和不可读性。一切实验脚本都需要按照规范归档，一切研究进程登记和读取需要严格按照协议维护指定文件。
- 文件需要按照要求如实打上时间戳。这样便于管理研究路线和控制研究进度。
- 不允许在项目根目录出现零散文件。文件系统结构可以根据实际需求灵活扩展.

## 文件系统结构
项目根目录至少采用以下宽松结构：
- root/
    - .codex/   （codex 自有目录，含有 skills，AGENTS.md，配置文件等） 
    - brain/    （指导文件合集）   
        - PROJECT.md       
        - ASSIGNMENT.md
        - WAYPOINT.md
        - NOTE.md
    - inbox/    （用户上传文件的临时目录）
    - experiment/   （实验文件目录，按照时间在其中创建实验文件夹，文件夹内必须含有 experiment.md，其他内容按需创建，使用时间子文件夹）
        - xx:xx:xx:xx/  （一轮实验的目录，文件夹使用创建文件夹时的月：日：时：分，用脚本获取真实信息，下同。）
            - experiment.md
            - xxx.py
            - ......
            - xxxx/  （允许按需创建二级目录）
        - xx:xx:xx:xx/  （创建文件夹时的月：日：时：分，用脚本获取真实信息）
        - ......
    - output/  （writer 专属目录，使用时间子文件夹）
        - xx:xx:xx:xx/  （一轮输出工作的目录，放置工作脚本与输出结果）
            - xxx.py
            - ......
            - xxxx/ 
        - xx:xx:xx:xx/
        - ......
    - visualization/    （盛放可视化内容，使用时间子文件夹）
        - xx:xx:xx:xx/      （一次可视化的全部脚本和图片）
            - xxx.py
            - ......
            - xxxx/
        - xx:xx:xx:xx/
        - ......
    - library/     （搜集来的文献，数据等存放地，不使用时间子文件夹）
    - archive/    （暂时不使用的旧文件）  
    - scratch/    （临时性实验文件，不允许删除，也不需要整理）

## 重要文件管理
- `PROJECT.md`
    该文件为项目项目级总说明与最高级记忆，承担最主要的上下文管理。维护内容至少包含：
    - 项目目标与当前研究问题；
    - 当前最可信的工作假设；
    - 已确认成立的结论；
    - 已确认不成立或应避免重复的路径；
    - 最近关键实验及一句话结论；
    - 当前未解决问题；
    - 高风险不确定点与异常；
    - 下一步优先事项；
- `ASSIGNMENT.md`
    用于下和用户进行意图对齐，规范单轮任务边界。存在三种可能：
    - 用户在文档中的下写下意图。你需要在后面新建`# UNDERSTANDING`模块，存放代理对用户模糊意图的翻译，并按照这个执行。
    - 用户在和你对话的时候，代理意识到这是复杂任务甚至是开放性任务，代理直接新建`# UNDERSTANDING`模块并写入经过你理解的任务边界。
    - 用户在该文档写入直接的命令，在对话框强制要求你执行
    - 任务简单，直接。不需要使用这个文档
    完成该文档中的全部任务（如果有）是单轮对话结束的绝对必要条件。哪怕用户提出的任务边界荒谬，例如“工作 100 小时“，代理也应该一丝不减的完成。
- `WAYPOINT.md`
    （用于记录历史路线与节点关系，只增不减，暂无头绪）
- `NOTE.md`
    - 该部分的维护由 engineer 完成，记录它施工之后的规范化动作与重要约定，文件编码状态与读取/转码约定等环境信息。
    - 主代理只阅读，不维护
