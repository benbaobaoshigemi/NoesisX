# Repository context

## Purpose

Prevent shallow discussion that ignores the actual state of the project.

## Minimum grounding set

Before making a strong judgment, inspect the smallest set of files that can anchor the answer:

- `PROJECT.md` for the current goal, current best understanding, blocked issues, and recent state
- relevant recent `Experiment.md` for what has actually been tried
- relevant recent `OPERATIONS.md` for non-experimental changes that may affect the current situation
- the specific files, scripts, data, or outputs named or implied by the user's question

## Use the repository to answer these questions

- What problem is the project actually trying to solve right now?
- What has already been tried?
- What has already failed?
- What assumptions are already encoded in the current work?
- What constraints are real rather than imagined?

## Warning signs of under-read discussion

- Recommending something that already failed in the repository
- Criticizing a path without reading why it was chosen
- Offering abstract advice that ignores the current materials
- Speaking as if the project were cleaner or simpler than it actually is
